# 场边教练叫醒策略：关键时刻低声提醒

## 1. 结论

Meetly 的场边教练不应该简单地“每 10 秒输出一条建议”。这会让它像弹幕或规则提示器，而不是一个懂现场的教练。

当前产品决定是：会话中如果已有转写、场边教练没有正在工作，就每 10 秒让它观察一次；但是否输出，交给模型自己判断。它可以输出一句可直接使用的话，也可以返回 `SILENT` 保持安静。

我们要做的是一套分层叫醒机制：

```text
持续转写
  -> 先写入会话上下文
  -> 强事件直接叫醒场边教练，比如新问题、沉默、长回答
  -> 没有强事件时，10 秒心跳做低优先级观察
  -> 场边教练决定：给可直接开口的话，或者保持安静
  -> Enter 仍然是用户主动要答案的强触发
```

这套机制借鉴 OpenClaw 和 NanoClaw，但不能照搬。OpenClaw 面向的是“长期在线的通用智能体”，NanoClaw 面向的是“消息队列驱动的异步智能体容器”。Meetly 的场景是实时面试/会议辅助，核心指标不是完成长任务，而是低延迟、少打扰、关键时刻有用。

## 2. 几个词的人话定义

| 词 | 人话解释 | 在 Meetly 里的意思 |
|---|---|---|
| 智能体 | 一个有目标、能看上下文、能决定下一步的 AI 执行者 | 我们的旁观者和未来办公助手 |
| 场边教练 | 不等用户提问，也会观察会话的人 | PI 运行时里的侧边智能体 |
| 心跳 | 定时检查一次，不代表一定输出内容 | 当前每 10 秒检查一次场边教练是否要工作 |
| 叫醒 | 真的让场边教练看上下文并运行一次 | 调 `runPiCoach({ trigger })` |
| 便宜判断 | 不用大模型，只用代码规则判断要不要叫醒 | 比如“有新问题”“用户沉默 8 秒”“刚刚已经提示过” |
| 保持安静 | AI 认为没必要打扰用户 | 返回 `SILENT`，界面不新增提示 |
| 预生成 | 用户还没按 Enter，先把可能要用的答案准备好 | 高置信度问题触发 `prefetch` |

## 3. 场边教练的业务定位

场边教练不是另一个聊天机器人，也不是“建议生成器”。它更像一个站在用户身边的 interview / meeting 教练。

它的目标不是每次听到问题就输出，而是持续判断：

```text
此刻用户最需要的下一句话是什么？
```

### 3.1 和 Enter 的分工

| 功能 | 定位 | 输出形态 |
|---|---|---|
| Enter | 用户主动求助 | 更完整的答案、解释、回答草稿 |
| 场边教练 | 系统主动低声提醒 | 一句开口语、纠偏、补强、收尾、或保持安静 |
| 转写区 | 事实记录 | 刚才说了什么 |
| 会后报告 | 复盘 | 表现、问题、改进 |

一句话区别：

```text
Enter：帮我回答这个问题。
场边教练：我现在可能需要你低声提醒我一下。
```

### 3.2 教练的思维模型

场边教练每次被叫醒后，应该按这个顺序思考：

```text
判断局势
  -> 判断风险
  -> 给最小帮助
  -> 或保持安静
```

**判断局势**

它先判断现场处于什么状态：

- 对方在提问；
- 用户正在回答；
- 用户卡住了；
- 用户答偏了；
- 用户说太散了；
- 对方在追问；
- 当前没有必要打扰。

**判断风险**

它再判断用户是否可能失分：

- 没有回答问题本身；
- 只有概念，没有个人经历；
- 说了很多但没有结论；
- 被追问带偏；
- 错过展示优势的机会；
- 沉默太久，需要开口救场。

**给最小帮助**

如果值得打扰，它只给最小、可直接使用的帮助：

- 一句开口语；
- 一个回答骨架；
- 一个补充角度；
- 一句自然收尾；
- 一个反问问题；
- 一句“别继续往这个方向讲”的纠偏。

差的输出：

```text
这是一个技术问题。先给一句明确结论，再补一个例子。
```

好的输出：

```text
可以先说：“我会先确认影响面，再从指标、链路和最近变更三个方向排查。”
```

场边教练越克制，用户越信任它。它不应该证明自己一直在工作，而应该在关键时刻有用。

## 4. OpenClaw 的规则是什么

OpenClaw 官方文档把心跳定义为“定期跑一次主会话里的智能体”。它不是另起一个智能体，也不是后台任务记录，而是调度器在合适的时间往主智能体会话里塞一个心跳轮次。

关键规则：

| 机制 | OpenClaw 做法 | 对我们的启发 |
|---|---|---|
| 定时 | 默认按配置间隔跑，比如 30 分钟 | Meetly 可以有定时观察，但间隔要短得多 |
| 是否调用大模型 | 调度本身是代码；真正心跳轮次会调用智能体/大模型，除非提前跳过 | 我们不能每次都直接调大模型，要先用规则挡一下 |
| 没事怎么办 | 智能体返回 `HEARTBEAT_OK`，系统把这类回复吞掉 | 我们用 `SILENT`，没价值就不显示 |
| 上下文控制 | 支持轻上下文、隔离会话、忙碌时跳过 | 我们也应该只给旁观者最近关键上下文，而不是每次塞全部 |
| 可见性 | 可以只跑不通知，也可以发通知 | 我们默认只显示短提示，不做大段报告 |
| 忙碌保护 | 可以在队列、任务、子智能体忙时跳过 | 我们已有 `in_flight` 跳过，后续要加更多规则 |

所以，OpenClaw 的“叫醒机制”不是一个神秘的大模型路由器。它的底层是代码调度和规则；调度命中后，才把这次心跳交给智能体判断要不要说话。

这点很重要：最好的主动式体验通常不是“AI 每隔几秒自己想一想”，而是“系统先判断有没有值得看的变化，再让 AI 做判断”。

## 5. NanoClaw 的机制是什么

NanoClaw 本地项目更像消息队列系统，不是 OpenClaw 那种主会话心跳。

它的核心规则是：

```text
消息进入数据库
  -> trigger = 0：只存上下文，不叫醒智能体
  -> trigger = 1：这是需要处理的消息，可以叫醒智能体
  -> host sweep 每 60 秒扫描有没有到期消息
  -> 有到期 trigger=1 消息，才唤起容器
  -> 容器里 runner 轮询消息
  -> 如果全是 trigger=0，继续等，不调用智能体
```

本地代码对应关系：

| NanoClaw 文件 | 关键点 |
|---|---|
| `/Users/zhujianye/study/nanoclaw/src/db/session-db.ts` | `countDueMessages` 只统计 `trigger = 1` 的消息 |
| `/Users/zhujianye/study/nanoclaw/container/agent-runner/src/poll-loop.ts` | 如果一批消息里没有 `trigger === 1`，就不调用智能体 |
| `/Users/zhujianye/study/nanoclaw/container/agent-runner/src/scheduling/task-script.ts` | 任务脚本可以返回 `wakeAgent=false`，提前跳过智能体 |
| `/Users/zhujianye/study/nanoclaw/src/host-sweep.ts` | 定时扫描、唤起容器、检查容器是否卡死 |

这对 Meetly 的启发更直接：

| NanoClaw | Meetly 可以怎么学 |
|---|---|
| `trigger=0` | 普通转写只进上下文，不叫醒旁观者 |
| `trigger=1` | 新问题、连续追问、沉默、答偏风险，才叫醒旁观者 |
| `wakeAgent=false` | 便宜判断发现没新信息，直接跳过 |
| host sweep | Meetly 的 10 秒检查器只负责“是否需要观察”，不直接等于调用 AI |
| 容器保活心跳 | Meetly 也需要日志判断旁观者是否卡住，但这和产品提示不是一回事 |

## 6. Meetly 现在的规则

当前相关代码：

| 文件 | 当前行为 |
|---|---|
| `src/app/useAutoAssist.ts` | 每个转写片段进入后，检测它像不像问题；像问题就触发旁观者，并且高置信度时预生成答案 |
| `src/app/usePiCoach.ts` | 会话中每 10 秒心跳一次；如果旁观者不忙且已有转写，就调用 PI 做低优先级观察 |
| `src/app/constants.ts` | `COACH_HEARTBEAT_MS = 10_000` |
| `src/runtime/piObserver.ts` | 调 PI 运行时，要求像现场教练一样输出可直接使用的短句，或 `SILENT` |

当前优点：

- 已经有主动观察，不按 Enter 也会触发旁观者。
- 已经有问题识别和预生成，用户按 Enter 时可以更快。
- 已经有 `in_flight` 保护，旁观者正在工作时不会重复叫醒。
- 已经有 `SILENT` 协议，旁观者可以选择不说话。
- 已经取消硬编码兜底文案，不再由代码塞“检测到问题/注意收束”这类模板句。

当前问题：

- “问题触发”和“心跳观察”的职责仍需要在 UI 和日志里表达得更清楚。
- 还没有“连续追问”“答偏风险”这些更像真人旁观者的触发器。
- 旁观者的输出质量还依赖模型是否理解“低声教练”定位，需要继续通过提示词和测试样例校准。
- UI 上看到的是输出结果，但看不到这次为什么被叫醒，后续排查仍然主要依赖本地日志。

## 7. Meetly 应该变成什么样

我们要把旁观者做成“面试现场的低声提醒者”，不是聊天机器人，也不是每次输出一大段的传统 AI。

推荐架构：

```text
转写缓冲区
  保存所有转写

叫醒策略
  用代码规则判断要不要叫醒旁观者

场边教练
  只在值得观察时运行

展示策略
  决定展示、吞掉、合并还是替换提示
```

### 7.1 叫醒规则

| 场景 | 便宜判断 | 是否叫醒旁观者 | 期望输出 |
|---|---|---|---|
| 识别到新问题 | 问句置信度高，且不是重复问题 | 是 | 给一句开口语或短答案骨架 |
| 连续追问 | 2 分钟内多个问题都围绕同一主题 | 是 | 提醒真实考察点，并给转向句 |
| 用户沉默 | 问题后 6-10 秒没有新回答 | 是 | 给一句可以立刻开口的话 |
| 用户回答很长 | 用户连续说了 20 秒以上 | 是 | 给一句自然收尾的话 |
| 答偏风险 | 最近回答和问题关键词重合低 | 是 | 给一句拉回问题本身的话 |
| 10 秒心跳 | 已有转写，旁观者空闲 | 是，低优先级 | PI 自己判断输出或 `SILENT` |
| 没有任何转写 | 会话刚开始还没有内容 | 否 | 不调用 AI |
| 旁观者正在工作 | `in_flight = true` | 否 | 记录跳过原因 |
| 刚刚有有效提示 | 最近几秒已有提示且无更强事件 | 仍可观察，但更倾向 `SILENT` | 避免刷屏 |
| 用户按 Enter | 明确主动请求答案 | 是，且优先级最高 | 直接给可说出口答案 |

### 7.2 触发优先级

```text
P0 用户主动：Enter
P1 强事件：新问题、高置信度追问
P2 现场风险：沉默太久、答偏、回答发散
P3 定时观察：10 秒到了，旁观者空闲，且会话已有转写
P4 会话维护：开始、结束、生成复盘
```

### 7.3 推荐的下一版流程

```text
每段转写完成
  -> 加入 transcriptHistory
  -> 计算 wake signals
     - hasNewQuestion
     - repeatedFollowup
     - silenceAfterQuestion
     - longAnswer
     - staleSinceLastCoach
  -> 如果 signal 足够强：叫醒旁观者
  -> 如果只是普通文本：只保存，不叫醒

每 10 秒
  -> 如果旁观者正在工作：跳过
  -> 如果还没有任何转写：跳过
  -> 如果有未处理的新问题/沉默/答偏风险：叫醒
  -> 否则以 fresh_context 叫醒一次
  -> PI 自己决定输出短句或 SILENT
```

## 8. “厉害”的地方应该在哪里

我们不需要一开始就做复杂多智能体。Meetly 真正厉害的地方应该是：它知道什么时候该说，什么时候不该说。

具体可以做成三层：

### 第一层：规则叫醒

代码负责快速判断，成本低、可控、可解释。

例子：

- 这段是否像问题；
- 距离上次新问题过了多久；
- 用户有没有开始回答；
- 这次回答有没有明显偏题；
- 旁观者上次看到的转写编号是多少。

### 第二层：旁观者判断

场边教练只在规则认为“值得看”时运行。它的职责不是写长答案，而是判断现场意义：

- 面试官真实意图是什么；
- 用户现在最该补哪一句；
- 这轮问题考察的是经验、原理、风险意识，还是沟通结构；
- 没有价值就返回 `SILENT`。

### 第三层：展示策略

即使旁观者输出了内容，也不一定全展示。

展示规则：

- 一次最多 1-2 句；
- 新提示比旧提示更重要时替换；
- 同一类提示短时间内合并；
- 用户按 Enter 时进入“答案模式”，优先显示可直接说出口的答案；
- 旁观者提示始终像朋友低声提醒，不像 Markdown 报告。

## 9. OpenClaw 最有价值的创新点

基于公开文档和本地参考代码，我认为 OpenClaw 最有价值的不是“有心跳”本身，而是这几个组合：

| 创新点 | 为什么重要 |
|---|---|
| 智能体长期在线 | 智能体不是一次性问答，而是有会话、有状态、有后续动作 |
| 本地/自有运行时 | 用户更能控制数据、工具、身份和接入方式 |
| 多通道接入 | 智能体可以接消息渠道、网页、命令行、工具，而不是只在一个聊天框里 |
| 身份和权限接入 | 不同人、渠道、工具有不同权限，智能体不是裸奔调用工具 |
| 心跳机制 | 即使没人发消息，智能体也能定期检查是否有事 |
| 忙碌与可见性控制 | 不是每次心跳都打扰用户，可以跳过、吞掉、只显示提醒 |
| 工具和任务模型 | 智能体能连接实际动作，而不只是输出文字 |

所以心跳是必要组件，但不是全部。OpenClaw 的核心更像“让智能体成为一个可运行、可治理、可接入现实世界的长期服务”。

## 10. 和 Manus 的差别

这里按公开产品定位做比较，不假设 Manus 内部实现细节。

| 维度 | Manus | OpenClaw / NanoClaw 风格 | Meetly 应该走的方向 |
|---|---|---|---|
| 核心场景 | 给一个目标，异步完成任务 | 搭一个可接入多渠道、多工具的智能体运行底座 | 实时对话辅助 |
| 工作方式 | 更像远程虚拟同事，自己规划执行 | 更像用户可控的智能体网关和运行时 | 更像现场旁观者和低声提醒者 |
| 时间尺度 | 分钟到小时 | 分钟、小时、天都可以 | 秒级到分钟级 |
| 交互重点 | 产出完整结果 | 消息、任务、工具、权限、运行状态 | 延迟、是否打扰、关键时刻提示 |
| 用户是否盯着 | 可以离开等结果 | 可离开，也可接消息 | 用户正在面试/开会，必须低干扰 |
| 最重要能力 | 规划和执行复杂任务 | 可治理的长期智能体基础设施 | 实时理解对话现场 |

Manus 的强点是“从目标到交付”，比如让它研究、整理、生成文件、做网站。Meetly 的强点不应该是复制 Manus，而是做它不擅长的现场低延迟上下文辅助。

换句话说：

```text
Manus：我给你一个目标，你去完成。
OpenClaw：我给智能体一个长期运行底座，让它接渠道、接工具、接权限。
Meetly：我正在跟人说话，你在旁边观察，关键时刻提醒我一句。
```

## 11. 落地计划

### P0：先把 10 秒心跳改成可解释的主动观察

新增一个前端策略模块，例如：

```text
src/app/coachWakePolicy.ts
```

它负责输出：

```ts
type WakeDecision =
  | { shouldWake: true; trigger: "heartbeat"; reason: string }
  | { shouldWake: false; reason: string };
```

需要跟踪：

- 上次旁观者看到的转写段编号；
- 上次旁观者有效输出时间；
- 最近一次问题候选；
- 最近一次用户沉默开始时间；
- 当前是否有预生成答案；
- 当前是否有旁观者调用正在进行。

验收标准：

- 还没有任何转写时，10 秒检查不调用 PI；
- 已有转写且旁观者空闲时，10 秒检查会触发一次低优先级观察；
- 有新问题但用户没按 Enter，旁观者会主动观察；
- 用户沉默时，旁观者能给短提示；
- 日志里能看到每次心跳是 `wake` 还是 `skip`，以及原因；
- 不影响 Enter 直接要答案。

### P1：把旁观者拆成几类观察任务

不要一上来多智能体，先做“一个旁观者，多种观察任务”：

| 观察任务 | 触发 |
|---|---|
| 问题意图 | 新问题或连续追问 |
| 回答质量 | 用户回答超过一定长度 |
| 沉默救场 | 问题后长时间没新回答 |
| 追问归因 | 多个问题围绕同一主题 |
| 会后复盘素材 | 会话结束时整理 |

### P2：再考虑更智能的叫醒器

等 P0/P1 稳定后，可以用一个更轻的判断器来替代部分规则：

- 小模型判断“这段有没有值得提醒的点”；
- 本地模型做低成本过滤；
- 场边教练只处理过滤后的重要事件；
- 长期记忆参与面试风格判断。

但现在不建议先上这个。原因很简单：实时面试里，稳定和可解释比“看起来智能”更重要。
