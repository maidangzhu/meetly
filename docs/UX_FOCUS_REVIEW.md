# Meetly 专注体验审查

## 1. 产品原则

Meetly 应该像一个安静、可靠的高压对话工具。界面要帮助用户保持心流，而不是让用户额外管理一个忙碌的仪表盘。

默认姿态：

```text
安静监听
  -> 只浮出此刻有帮助的信息
  -> 原始上下文可查看，但保持次要
  -> 避免调试语言和内部状态
```

## 2. 屏幕区域职责

| 区域 | 用户理解 | 交互规则 |
|---|---|---|
| 顶部悬浮岛 | 开始/结束会话、打开面板、隐身状态 | 紧凑稳定，不跳动 |
| 回答建议 | 用户主动求助 | Enter 后最高优先级 |
| 场边教练 | 主动、低噪音帮助 | 只有真的有内容时才可见 |
| 转写 | 事实记录和上下文 | 可滚动，保持次要 |
| 设置 | 配置和权限 | 不打断实时会话 |

## 3. 场边教练行为

场边教练不应该因为自己正在后台观察就显示活动状态。

好的行为：

- 后台观察时保持不可见。
- 只有出现可直接使用的一句话时才显示。
- 避免“检测到问题”“回答过长”这类标签。
- 对可能返回 `SILENT` 的调用，不显示闪烁转圈。
- 消息足够短，用户不用脱离当前对话也能读完。

坏的行为：

- 每次心跳都闪一下。
- 显示内部置信度、触发器、供应商状态。
- 用泛泛建议塞满侧栏。
- 让用户分不清是 UI 坏了还是正在思考。

## 4. 视觉方向

界面应该克制、专业、偏工具属性：

- 信息密度可以高，但必须可扫读。
- 不做营销式大卡片和装饰性布局。
- 悬浮岛和面板区域尺寸稳定。
- 绿色只用于录音中、已准备等少数状态。
- 实时界面不出现调试文案。
- 回答建议、场边教练、转写之间必须有明确主次。
- 各滚动区域独立，不能互相挤压。

## 5. 交互方向

P0 优化：

- 后台观察不显示思考状态，直到真的有文字流出。
- 用户可见命名统一为“场边教练”。
- 移除实时面板里的调试计数和英文状态。
- 把预生成、供应商状态转换成简短中文状态。

P1 优化：

- 视觉上区分“回答建议”和“场边教练”。
- 场边教练保持固定可见但不抢注意力，转写保持次要。
- 空状态表达当前状态，但不写教程式说明。
- 重新评估面板宽度和断点，避免教练栏过窄。
- 为当前建议、最新教练提醒、转写建立更清晰层级。

P2 优化：

- 增加专注实时模式，只保留悬浮岛和最新教练提醒。
- 会话结束后进入复盘模式，展示报告和关键片段。
- 后续再接入问屏幕/上下文模式，不能挤占面试主界面。
